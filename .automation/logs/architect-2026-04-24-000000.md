## Task decomposed

**REP-500** — extract MLX to separate SPM target so test compile doesn't link MLX

**Priority**: P0

## Why REP-500 was selected

Two open P0 L-tasks exist: REP-500 and REP-285. Both target the same root cause (MLX cold-build blocking `swift test`), but REP-500 is far more complete: it specifies the exact target structure, file-move paths, import changes, build.sh updates, and validation order. REP-285 scopes only to `Package.swift` — which is insufficient because SwiftPM requires non-overlapping source directories per target; you cannot carve MLX out of `Sources/ReplyAI/` without physically moving files to a separate directory. REP-285 is therefore architecturally incomplete as written and would fail if attempted. REP-500 supersedes it.

REP-500 is also the highest-impact unblocked task in the backlog: the MLX cold-build time (45–90 min) is the root cause of 20+ wip branches stranded awaiting human `swift test` verification. Fixing this would enable the merger agent to drain the wip queue automatically, multiplying automation throughput across the whole system.

---

## Chain structure and rationale

The decomposition follows the **refactor-by-structural-boundary** pattern: one sub-task per logical layer of the refactor.

| Sub-task | Role | Can compile after? |
|---|---|---|
| REP-501 | File moves + Package.swift rewrite | No (missing cross-module imports) |
| REP-502 | Cross-module imports in MLXDraftService + ReplyAIApp | Partially (app compiles, tests don't) |
| REP-503 | Test file import rename | Yes (full build should be green) |
| REP-504 | build.sh resource paths + dep-graph validation | Yes (final build confirmation) |
| REP-505 | Human: `swift test` + `./scripts/build.sh debug` + merge | N/A (human gate) |

### Why split at these boundaries?

- **REP-501 is the largest single unit** (~60–80 file moves + Package.swift rewrite). The file moves and Package.swift rewrite MUST happen atomically in the same commit — if Package.swift is rewritten before files move, the build fails immediately with "no source files found"; if files move before Package.swift is updated, SwiftPM can't locate them at all. Combining them in one M-effort commit keeps the wip branch in the most understandable intermediate state.

- **REP-502 is tiny but logically distinct** from REP-503. MLXDraftService.swift and ReplyAIApp.swift need imports because they are in *different* SPM targets; the test files need renaming because the *target name* changed. These are different error classes — keeping them separate helps a worker debug if one compiles and the other doesn't.

- **REP-503 is mechanical** — a sed one-liner across all test files. Worker needs no architectural judgment; pure substitution. Keeping it separate from REP-504 ensures the build.sh change doesn't get mixed with test-import changes, making each commit's diff easier for human reviewers to understand.

- **REP-504 + REP-505 separate build.sh maintenance from human verification**. The build.sh path fix is small but critical (without it, the bundled .app won't include Info.plist / entitlements / fonts). The dep-graph validation (`swift package show-dependencies`) provides machine-readable proof that the test target no longer pulls MLX. Human then confirms `swift test` passes end-to-end.

## Alternative decompositions considered and rejected

### Alternative A: Combine all 5 steps into 1 commit

Rejected. Violates the 10–30 min per sub-task budget. The file-move step alone is ~10 min; doing all import changes and build.sh on top would push a single worker fire beyond 13 min. More importantly, one massive commit is opaque to the human reviewer.

### Alternative B: Do NOT move MLXDraftService.swift; use Package.swift `exclude:` instead

Rejected. SwiftPM prohibits two targets from having overlapping source paths. Even with `exclude: ["Services/MLXDraftService.swift"]` on ReplyAICore, you cannot define a second target pointing at the same `Sources/ReplyAI/` path. The file MUST move to a new directory. This is why REP-285's "Package.swift only" scope is unachievable without file moves.

### Alternative C: Separate sub-tasks for each file's import fix (one task per file)

Rejected. MLXDraftService.swift and ReplyAIApp.swift are only 2 files, both needing cross-module imports for the same reason. Splitting them into 2 sub-tasks adds overhead (2 PRs, 2 planner cycles) with no real benefit. Combined they are well under 30 min.

### Alternative D: Move Resources/ in a separate sub-task

Rejected. Resources/ must be co-located with the Swift source move to keep the build.sh update meaningful. Moving Resources/ separately creates an intermediate state where Package.swift expects `Sources/ReplyAICore/Resources/` but the bundler still references the old path — two broken states instead of one clean intermediate.

## Risks

1. **The file-move step is the most dangerous**. Using `git mv` preserves history, but if a worker accidentally uses `cp` + `rm` instead of `git mv`, git history is lost. The success criteria for REP-501 explicitly checks `git diff --stat HEAD` to confirm moves (not copies).

2. **MLX cold-build constraint means REP-502 and REP-503 can only do `swift build`, not `swift test`**. The human gate (REP-505) is load-bearing. If REP-505 is skipped or bypassed, a broken build could land on main. The `claimed_by: human` on REP-505 should prevent automation from claiming it.

3. **`ReplyAIApp.swift`'s `@main` attribute**. SwiftPM requires exactly one `@main` in the executable target. Moving it to a separate directory changes the executable product source — the `Package.swift` rewrite must explicitly set `executableTarget(name: "ReplyAI", path: "Sources/ReplyAIApp")`. Failing to set the path would cause SPM to look in `Sources/ReplyAI/` (now deleted) and error.

4. **Resources path**. If `scripts/build.sh` misses any `Sources/ReplyAI/Resources/` reference, the .app bundle will silently omit Info.plist or entitlements, causing a runtime crash or macOS refusal to launch. REP-504's success criteria require `grep "Sources/ReplyAI/" scripts/build.sh` to return empty.

5. **REP-285 orphan**. REP-285 (P0, L, open) was not decomposed this cycle. It is architecturally infeasible as scoped (Package.swift only, no file moves). A follow-up architect fire should update REP-285 to `claimed_by: human` with `blocker: "superseded by REP-500 decomposition; Package.swift-only approach infeasible due to SPM source-path overlap constraint"`.

## Expected outcome

After REP-505 merges, `swift test` on a fresh clone should drop from 45–90 min (cold) to <5 min (cold) because the test target will no longer pull in the MLX C++ dependency chain. This unblocks the replyai-merger to drain the entire wip queue automatically, removes the cold-build warning from AGENTS.md/worker.prompt, and restores the 13-min worker budget to full productivity.
