## Notes

- FTS5 doesn't have `INSERT OR REPLACE` semantics keyed by application columns, so upsert is DELETE WHERE thread_id = ? + INSERTs under BEGIN/COMMIT. Keeps it atomic against a concurrent `search`.
- Chose a `didSeedSearchIndex` flag on the VM over plumbing "is this an incremental sync" into the channel layer. Simpler — the VM already owns the decision about when it's initial-vs-subsequent.
- The task scope referenced an `InboxViewModel.handleIncomingMessages(_:)` method that doesn't exist — the real code path is `syncFromIMessage`, which fires from the watcher. Wiring in that method achieves the same outcome.
