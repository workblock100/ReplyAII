## Next-fire hints

- **Chronic hang in `InboxViewModelTests.testConcurrentSyncCallsDoNotOverlap`**
  blocks every full `swift test` run. Stack trace via `sample` shows
  `XCTWaiter waitForExpectations` waiting in mach_msg_overwrite — the
  test uses `BlockingMockChannel.enteredStream` (AsyncStream) to wait
  for sync1 to enter the blocking section before issuing sync2. Hypothesis:
  on this Apple Silicon + macOS 26.3 + Xcode 16.0 environment, when
  AsyncStream's continuation yields before the for-await consumer iterates,
  the value is buffered fine — but XCTest's bridge to async test methods
  creates a top-level XCTWaiter that times out only if a timeout is set,
  and this test sets none. Net effect: when scheduling lands wrong it
  deadlocks. Not introduced by anything merged this fire. The
  `f46f496 tests: replace Task.yield polling with enteredStream wait in
  IsSyncing test` branch was the previous fire's attempt at addressing
  the related IsSyncing flake — this one is the same shape but on a
  different test. Worth a focused fix in a future fire: either set an
  explicit `testTimeout` on `XCTestCase` or refactor the test to await
  a Task with a timeout-bounded waitForCompletion helper.

  RulesTests (preconditions assert "thread starts unpinned" but pinned
