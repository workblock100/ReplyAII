## P0 finding (diagnosed; candidate fix on wip; verification deferred)

**`swift test` SIGSEGVs at ~358 of 1321 tests.** Reproduces on clean
main HEAD `9d8a897`. Bisect proved older sha `0b7290f` ran 1299 tests
with zero crashes — the regression was introduced by one of the
13 test-only commits between `0b7290f` and `9d8a897`. Each suite passes
in isolation (`swift test --filter <SuiteName>` always exits 0); the
crash only happens in the full sequenced run, so it's cross-suite state
leakage not a faulty test in itself.

**Crash signature**: process exits with signal 11 between `FixturesTests`
(passes 40/40) and `IMessageChannelIsReadTests` (passes 1/1 in
isolation). The error message interleaves with the test runner's
suite-name printing, indicating the crash happens during a
deinit/teardown phase of an earlier suite, not inside a running test.

**Strong suspect**: commit `d290dc5` `tests: ReplyAIWindowSummoner —
pin notification fallback + fast-path window match` added
`testSummonFastPathMatchesWindowTitledInbox` which constructs an
`NSWindow` inside the test body, runs `ReplyAIWindowSummoner.summon()`
on `@MainActor`, drains the run loop with
`RunLoop.main.run(until:0.05)`, and relies on `defer { window.close() }`
for cleanup. Untested hypothesis: the `NSWindow` lingers in
`NSApp.windows` after teardown, and a later AppKit-touching test (or
the xctest harness itself reaching a quiescent state) trips on the
zombie window during dealloc, segfaulting the runner. AppKit objects
created outside an `NSApplication.shared.run()` event loop are a
known crash source in headless `xctest` processes.

**Verification approach for next fire**:
1. Locally revert just `Tests/ReplyAITests/GlobalHotkeyContractTests.swift`
   to its `d290dc5~1` content (i.e. before the new tests).
2. Run `swift test`. If exit code 0 and tests run >800, the
   `NSWindow`-creating tests are the cause — confirmed.
3. Fix: either gate the two new `@MainActor` tests behind a
   `#if !RUNNING_IN_HEADLESS_XCTEST` guard (env var check), or
   restructure the test to not construct a real NSWindow (use a
   protocol seam in `ReplyAIWindowSummoner` that takes a
   `[(title: String)]` array instead of reaching into `NSApp.windows`),
   or simply mark them `XCTSkipIf(NSApp.windows.isEmpty …)` so the test
   no-ops in headless. The protocol-seam fix is preferred because it
   pins more of the contract.

This fire did not implement the fix because diagnosis consumed the bulk
of session time and the fix needs careful design (the window-summoner
test is checking real NSApp behavior, so the seam needs care to keep
the assertion meaningful).
