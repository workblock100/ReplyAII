## Gotcha for next fire — workspace contention

Two autopilot fires were running concurrently in the same `~/.cache/replyai-autopilot`
checkout. The second fire's startup `git reset --hard origin/main` + `git clean -fd`
silently wiped this fire's uncommitted edits. **The persistent workspace assumes one
fire at a time**, which the cron schedule respects only if each fire stays inside its
60-min window. This fire ran 113 min because of `swift test` taking ~38 min (CoreData
NSXPCConnection retry storms from the ContactsResolver carve-out add ~3s × N tests of
overhead) plus build-lock waits. Mitigation options for future fires:

1. **Always commit code before running `swift test`.** Even if the test will retroactively
   guard the merge, committing first makes the work survive any future workspace reset.
   `git stash` would also work.
2. **Shorten the swift-test surface used for self-validation.** `--filter <NewSuite>` is
   sufficient for the kind of diff this fire was doing (one struct + 5 tests, no
   touched call sites). Reserve full-suite runs for branches that touch shared code.
3. **Apply REP-P9 (ContactsResolverTests `XCTSkipIf`) before anything else** — the
   ~38-min full-test runtime is largely those 21 tests retrying NSXPCConnection 8
   times each at 3s per retry. Skipping them headlessly would bring full-suite runtime
   under ~5 min and remove a major source of fire-budget overrun.
