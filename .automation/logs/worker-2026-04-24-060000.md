## Summary

Bug fix in the shipped REP-235 notification path. `applyIncomingNotification` was
matching only by senderHandle heuristic (suffix match or name equality), which failed
when a thread's chatGUID format didn't match the handle exactly — causing duplicate
thread entries on repeated notifications.

Fix:
1. `NotificationCoordinator.willPresent` now extracts `CKChatIdentifier` (primary) or
   `CKChatGUID` (fallback) from `content.userInfo` and passes it through
   `handleIncomingNotification(chatGUID:)`.
2. `InboxViewModel.applyIncomingNotification` accepts `chatGUID: String?` (default nil
   for backward compat). When non-nil, does exact match on `thread.chatGUID` instead of
   the heuristic. Falls back to heuristic only when chatGUID is nil.

5 new tests:
- `testChatGUIDExtractedFromCKChatIdentifier`
- `testChatGUIDFallsBackToCKChatGUID`
- `testIncomingNotificationWithMatchingGUIDUpdatesExistingThread`
- `testIncomingNotificationWithUnknownGUIDCreatesNewThread`
- `testIncomingNotificationWithNilGUIDCreatesNewThread`
